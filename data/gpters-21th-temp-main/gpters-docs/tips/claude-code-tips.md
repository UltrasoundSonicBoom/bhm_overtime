# Claude Code 실전 팁 모음 (47선)

> 초보자부터 중급자까지, 바로 써먹을 수 있는 팁을 카테고리별로 정리했습니다.

---

## 1. 시작하기 (7개)

1. **설치는 한 줄이면 끝.** 터미널에서 `npm install -g @anthropic-ai/claude-code` 실행
2. **프로젝트 폴더에서 `claude` 입력.** 반드시 프로젝트 루트(최상위 폴더)에서 실행해야 전체 코드를 이해함
3. **처음 실행하면 `/init` 부터.** 프로젝트를 분석해서 CLAUDE.md 초안을 자동으로 만들어줌
4. **CLAUDE.md = 프로젝트 설명서.** 코딩 스타일, 빌드 명령어, 주의사항을 적어두면 매 세션마다 자동으로 읽음
5. **CLAUDE.md는 짧게 유지.** 너무 길면 중요한 규칙을 무시함. "이거 빼면 Claude가 실수할까?" 기준으로 정리
6. **VS Code 확장도 있음.** 확장 마켓에서 "Claude Code" 검색 → 설치 → Cmd+Shift+P → "Claude Code" 선택
7. **데스크톱 앱도 있음.** 여러 세션을 나란히 열어서 병렬 작업할 때 편리

---

## 2. 프롬프트 팁 (8개)

1. **"~해줘"보다 "~를 ~방식으로 해줘"가 정확.** 예: "테스트 추가해줘" → "foo.py에 로그아웃 상태 엣지케이스 테스트를 mock 없이 추가해줘"
2. **파일 경로를 명시하면 엉뚱한 파일 수정 방지.** "src/auth/login.ts를 수정해줘"처럼 구체적으로
3. **파일 경로를 직접 입력하거나 `@파일명`으로 참조.** `src/utils/helper.ts`를 언급하거나 `@src/utils/helper.ts`로 참조하면 Claude가 직접 읽음
4. **에러 메시지는 그대로 붙여넣기.** 요약하지 말고 전체를 복사해서 주면 정확한 원인 파악 가능
5. **기존 패턴을 알려주기.** "HotDogWidget.php 패턴을 참고해서 CalendarWidget을 만들어줘"
6. **스크린샷도 붙여넣기 가능.** UI 버그를 설명할 때 스크린샷을 드래그앤드롭하면 시각적으로 이해
7. **파이프로 데이터 전달.** `cat error.log | claude -p "분석해줘"` 하면 로그 내용을 바로 분석
8. **모호한 프롬프트도 가끔은 유용.** "이 파일에서 개선할 점이 뭐야?" 같은 열린 질문으로 예상 못한 인사이트 발견

---

## 3. Plan Mode (6개)

1. **Shift+Tab으로 Plan Mode 진입.** 파일을 수정하지 않고 탐색/계획만 하는 안전한 모드
2. **복잡한 작업은 반드시 Plan Mode 먼저.** 여러 파일을 건드리거나 구조를 바꿀 때 필수
3. **탐색 → 계획 → 실행 → 커밋 순서를 지키기.** 이 4단계 워크플로우가 가장 실수가 적음
4. **계획을 승인하기 전에 수정 대상 파일 목록 확인.** "어떤 파일을 수정할 건지 알려줘"로 먼저 확인
5. **간단한 작업은 Plan Mode 건너뛰기.** 오타 수정, 변수 이름 변경 같은 건 바로 해도 됨
6. **큰 기능은 인터뷰부터.** "이 기능에 대해 인터뷰해줘. 내가 놓친 부분을 질문해줘"로 시작하면 완성도 올라감

---

## 4. 파일 작업 & Git (7개)

1. **"이 파일 읽고 설명해줘"로 코드 이해.** 새 프로젝트 온보딩할 때 시니어 개발자한테 묻듯이 질문
2. **git diff 확인하는 습관.** 작업 후 "변경 사항 보여줘"로 뭐가 바뀌었는지 항상 체크
3. **커밋도 Claude에게 맡기기.** `"변경 사항을 설명하는 메시지로 커밋해줘"` 하면 알아서 커밋
4. **PR 생성도 한 줄이면 끝.** `"PR 만들어줘"` — 브랜치 생성, 푸시, PR 오픈까지 자동
5. **git 히스토리로 코드 배경 파악.** "이 함수의 git history를 보고 왜 이렇게 됐는지 설명해줘"
6. **Esc 두 번으로 되돌리기.** Claude가 파일을 잘못 수정했으면 체크포인트로 복원 가능
7. **체크포인트는 세션을 닫아도 유지됨.** 나중에 돌아와서도 이전 상태로 복원할 수 있음

---

## 5. 디버깅 (7개)

1. **에러를 그대로 붙여넣기 + "고쳐줘".** Claude가 원인을 추적하고 수정까지 해줌
2. **"근본 원인을 해결해줘, 에러를 숨기지 마"라고 명시.** 안 그러면 try-catch로 감싸버리는 경우 있음
3. **테스트를 검증 수단으로 제공.** "이 테스트가 통과하도록 고쳐줘" — Claude가 스스로 확인 가능
4. **"테스트 작성 → 실행 → 실패하면 수정"을 한 프롬프트에.** 예: "auth 모듈 테스트 작성하고, 실행하고, 실패하면 고쳐줘"
5. **증상과 위치를 같이 알려주기.** "로그인이 세션 만료 후 실패함. src/auth/ 토큰 리프레시 쪽 확인해줘"
6. **로그 파일을 파이프로 넘기기.** `tail -f app.log | claude -p "이상한 게 있으면 알려줘"`
7. **서브에이전트로 조사 위임.** "서브에이전트를 써서 인증 시스템의 토큰 갱신 방식을 조사해줘" — 메인 컨텍스트를 오염시키지 않음

---

## 6. 스킬 & 커맨드 (7개)

1. **스킬 = 반복 작업의 자동화 레시피.** `.claude/skills/` 폴더에 SKILL.md 파일로 만듦
2. **SKILL.md에 단계별 워크플로우 작성.** Phase 1, 2, 3... 순서로 Claude가 따라갈 절차를 적음
3. **`/스킬이름`으로 실행.** 예: `/fix-issue 1234` → GitHub 이슈 확인 → 코드 수정 → PR 생성
4. **`$ARGUMENTS`로 인자 받기.** SKILL.md에 `$ARGUMENTS`를 넣으면 `/스킬이름 값`의 "값"이 들어감
5. **`disable-model-invocation: true` 옵션.** 부작용이 있는 스킬은 수동 호출만 허용하도록 설정
6. **커스텀 서브에이전트도 가능.** `.claude/agents/`에 전문 리뷰어, 보안 검토 담당 등을 정의
7. **Hooks로 자동 실행 보장.** "파일 수정 후 항상 eslint 돌려줘"를 CLAUDE.md가 아닌 Hook으로 설정하면 100% 실행됨

---

## 7. 실전 생산성 팁 (7개)

1. **`/clear`를 자주 쓰기.** 작업 주제가 바뀌면 컨텍스트를 리셋. 오래된 대화는 성능을 떨어뜨림
2. **두 번 수정해도 안 되면 `/clear` 후 다시 시작.** 실패한 시도가 쌓이면 Claude가 더 혼란스러워함
3. **`claude --continue`로 이어서 작업.** 터미널을 닫았다가 다시 열어도 이전 대화 이어가기 가능
4. **`claude --resume`으로 과거 세션 선택.** 이전 대화 목록에서 원하는 세션을 골라 이어가기 가능
5. **`/compact`로 컨텍스트 압축.** 대화가 길어졌을 때 `/compact API 변경에 집중`하면 핵심만 남김
6. **병렬 세션으로 생산성 배가.** 한 세션은 코드 작성, 다른 세션은 리뷰 — 서로 다른 관점에서 검증
7. **비대화 모드로 자동화.** `claude -p "프롬프트"` 형태로 CI 파이프라인, pre-commit 훅, 스크립트에 통합 가능

---

> **핵심 원칙 3가지:**
> 1. **검증 수단을 함께 주기** — 테스트, 스크린샷, 예상 결과를 제공하면 Claude가 스스로 확인
> 2. **컨텍스트 관리하기** — `/clear`와 서브에이전트로 대화창을 깨끗하게 유지
> 3. **탐색 → 계획 → 실행** — 바로 코딩하지 말고 Plan Mode로 먼저 파악
