## 한 달 동안 Claude Code로 개발해본 소감

약 한 달간 Claude Code를 실제 개발 업무에 적용해보면서 느낀 점들을 정리해보았습니다.  
결론부터 말씀드리면, **도구 자체보다 이를 어떻게 사용하는지가 훨씬 중요하다** 는 것을 다시 한 번 체감한 시간이었습니다.

### Context engineering의 중요성

Claude Code를 사용하면서 가장 크게 느낀 부분은 context engineering의 중요성이었습니다.  
맥락이 잘 정리되어 있을 때는 결과물의 품질도 자연스럽게 올라가지만,  
맥락이 흐트러지는 순간 모델의 한계가 바로 드러났습니다.

특히 프론트엔드에서 모든 파일을 한 번에 참조하도록 구성했을 때  
context가 감당하지 못하고 터지는 현상을 여러 번 경험했습니다.  
“다 알고 있으니 잘 처리해주겠지”라는 기대는 현실적이지 않았습니다.

### Monorepo 구성과 작업 분리

이 문제를 해결하기 위해 monorepo 구조로 전환하여  
각 repo 단위로 작업이 가능하도록 구성했습니다.

이렇게 하니 context 범위가 자연스럽게 제한되었고,  
Claude Code도 훨씬 안정적으로 동작하는 것을 확인할 수 있었습니다.

결국 AI 역시 사람이 일하듯,  
**일하기 좋은 환경을 만들어주는 것이 중요하다** 는 인상을 받았습니다.

배포 상황에서도 버젼을 컨트롤하고 운영하기 좋았던 경험이 있습니다.

### CLAUDE.md는 간결할수록 좋습니다

[CLAUDE.md](http://claude.md/) 는 길게 작성하는 것이 반드시 좋은 결과로 이어지지는 않았습니다.  
반드시 지켜야 할 규칙과 전제만 간결하게 정리하는 편이 오히려 효과적이었습니다.

설명을 많이 덧붙일수록  
모델 입장에서는 오히려 노이즈로 작용하는 경우가 많았습니다. 모델 성능이 충분히 좋아져서 최대한 짧게 100줄 이내로 줄이고자 계획하고 있습니다.

### 세분화된 계획의 필요성

계획은 가능한 한 잘게 나누는 것이 필수적이라고 느꼈습니다.  
이 부분은 사람이 일하는 방식과 크게 다르지 않습니다.

- 한 번에 큰 단위를 요청하면 결과가 흔들리고
- 작은 단위로 나눌수록 성공 확률이 높아집니다

또한 구현 방향을 작업 중간에 변경하게 되면,  
그동안 사용한 토큰이 그대로 낭비되는 경우가 많았습니다.  
체감상 비용과 시간 모두에서 손실이 적지 않았습니다.

### 팀 내 전파의 현실

팀 내 전파는 예상했던 대로 쉽지 않았습니다.  
“유용하다”는 점을 강조하며 설득하는 방식은 생각보다 효과가 크지 않았습니다.

오히려

> 사용하지 않으면 일이 더 번거로워지는 구조

를 만드는 편이 훨씬 현실적인 접근이라는 생각이 들었습니다.

### 바이브 코딩에 대한 생각

최근에는 바이브 코딩을 장려하는 방향이 더 적합하다고 느끼고 있습니다.  
필요성을 느낀 팀들이 각자 기능 단위의 프로토타입을 만들어오고,  
이를 병합하는 형태로 작업이 진행된다면 이상적인 흐름이 될 것 같습니다.

완벽한 설계보다는  
**빠르게 만들어보고, 조정하고, 합치는 방식** 이  
AI 기반 개발 환경에는 더 잘 맞는 접근이라고 생각합니다.
