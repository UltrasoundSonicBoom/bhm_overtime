# 급여 명세서 분석기 (Payroll Parser) 로직 설명

## 1. 개요

본 `PayrollParser`는 Excel(.xlsx, .xls) 및 CSV(.csv) 형식의 급여 명세서 파일에서 직원 정보, 급여 지급 항목, 공제 항목, 급여 요약 정보 등을 자동으로 추출하고 구조화하기 위해 설계되었습니다.

주요 목표는 다양한 급여 명세서 양식에 대응하는 것이지만, 특히 **고정된 행/열 구조를 가진 명세서에 대해 높은 정확도**를 제공하는 데 중점을 둡니다. 이는 사용자의 성공적인 수동 파싱 경험에서 확인된, 명세서 내 항목 이름과 금액 간의 명확한 위치적 연관성을 최대한 활용하는 방식입니다.

## 2. 핵심 파싱 전략: 고정 블록 추출 (`extractFixedBlockItems`)

이 전략은 `PayrollParser`의 가장 우선적이고 정확도가 높은 파싱 방식입니다. 특정 키워드(앵커)를 기준으로 급여 및 공제 항목 블록의 위치를 파악하고, 그 내부에서 항목 이름과 금액을 1:1로 매핑합니다.

### 2.1. 앵커(Anchor) 기반 위치 탐색

-   파서는 먼저 파일 내에서 주요 앵커 텍스트를 검색합니다.
    -   **급여 항목 앵커**: `SALARY_ANCHOR_TEXT` (예: "기본기준급")
    -   **공제 항목 앵커**: `DEDUCTION_ANCHOR_TEXT` (예: "소득세")
-   이 앵커 텍스트가 발견된 셀의 좌표(행, 열)가 해당 블록 파싱의 기준점이 됩니다.

### 2.2. 상대적 좌표계 및 블록 구조 정의

앵커 셀의 위치를 기준으로, 항목 이름들이 모여있는 "이름 블록"과 금액들이 모여있는 "금액 블록"의 범위가 상대적으로 결정됩니다. 이 구조는 다음과 같은 내부 설정 상수들에 의해 정의됩니다:

-   `SALARY_NAME_ROW_COUNT` / `DEDUCTION_NAME_ROW_COUNT`: 각 앵커로부터 시작하여 항목 이름이 몇 개의 행에 걸쳐 있는지 정의합니다. (예: 급여 이름 6행)
-   `SALARY_AMOUNT_ROW_OFFSET` / `DEDUCTION_AMOUNT_ROW_OFFSET`: 이름 블록의 시작 행으로부터 금액 블록의 시작 행까지의 간격(행 수)을 정의합니다. (예: 급여 이름 시작 행 + 6행 = 급여 금액 시작 행)
-   `SALARY_AMOUNT_ROW_COUNT` / `DEDUCTION_AMOUNT_ROW_COUNT`: 금액 블록 자체가 몇 개의 행으로 이루어져 있는지 정의합니다. (항목 이름의 각 행이 금액 블록 내에 대응하는 행을 갖는지 확인하는 데 사용됩니다.)
-   `SALARY_COL_COUNT` / `DEDUCTION_COL_COUNT`: 앵커가 위치한 열로부터 시작하여, 항목 이름과 금액을 스캔할 최대 열의 개수(너비)를 정의합니다.

### 2.3. 정확한 1:1 매핑

1.  파서는 앵커를 기준으로 정의된 "이름 블록" 내의 각 셀을 순차적으로 탐색합니다.
2.  `isSalaryItem` 또는 `isDeductionItem` 메소드(내부적으로 정규식 패턴 사용)를 통해 해당 셀의 텍스트가 유효한 항목 이름인지 확인합니다.
3.  유효한 항목 이름이 `이름블록[현재_행_오프셋][현재_열_오프셋]` 위치에서 발견되면, 파서는 **정확히 동일한 상대적 위치**인 `금액블록[현재_행_오프셋][현재_열_오프셋]`에서 해당 항목의 금액을 찾습니다.
    -   `현재_행_오프셋`: 이름 블록 내에서의 현재 행의 상대적 위치 (0부터 `NAME_ROW_COUNT - 1`까지).
    -   `현재_열_오프셋`: 앵커 열로부터의 현재 열의 상대적 위치 (0부터 `COL_COUNT - 1`까지).
4.  **빈 금액 처리**: 만약 금액 블록의 해당 셀이 비어 있거나, `isAmount` 메소드(숫자와 콤마만 허용)로 검사했을 때 유효한 금액 형식이 아니면, 해당 항목의 금액은 **'0'으로 처리**됩니다. 절대 주변의 다른 셀에서 금액을 가져오지 않아 데이터의 정확성을 보장합니다.
5.  **0원 항목 포함**: 항목 이름이 유효하다면, 그에 해당하는 금액이 0원이더라도 결과에 포함됩니다.

### 2.4. 항목 고유성

추출된 모든 항목은 최종적으로 "항목 이름 + 해당 항목이 원본 파일에서 발견된 열(column)의 인덱스"를 조합한 고유 키를 기준으로 필터링되어, 중복된 정보가 결과에 포함되는 것을 방지합니다.

### 2.5. 실행 조건

-   `extractSalaryItemsFromExcel` 및 `extractDeductionItemsFromExcel` 메소드 내에서, 각각의 앵커 텍스트가 성공적으로 발견되고, 이를 통해 `extractFixedBlockItems` 메소드가 하나 이상의 유효한 항목을 추출했을 경우, 이 결과가 최종적으로 사용됩니다.
-   만약 이 고정 블록 추출 전략이 실패하거나 유의미한 결과를 내지 못하면, (현재 주석 처리된) 다음 단계의 동적 파싱 전략으로 넘어갈 수 있습니다.

## 3. 기타 주요 기능

### 3.1. 직원 정보 추출 (`extractEmployeeInfoFromExcel`)

-   명세서 상단(보통 첫 10행 이내)에서 "개인번호", "성 명", "소 속", "직 종", "급여연차", "입사년월"과 같은 레이블을 정규식을 사용하여 검색합니다.
-   레이블이 발견되면, 주로 해당 레이블 우측의 셀(들)에서 값을 가져옵니다.
-   날짜 형식(예: 입사년월)은 `YYYY-MM-DD` 형식으로 정규화하려고 시도하며, Excel의 숫자 날짜 형식도 처리합니다.

### 3.2. 메타데이터 추출 (`extractMetadataFromExcel`)

-   명세서 상단(보통 첫 5행 이내)에서 "급여 기간"(예: "2024년도 12월분") 및 "급여지급일"과 같은 정보를 추출합니다.
-   날짜는 `YYYY-MM-DD` 형식으로 정규화합니다.

### 3.3. 요약 정보 추출 및 검증 (`extractSummaryFromExcel`, `calculateSummary`)

-   **직접 추출**: 명세서 하단(보통 마지막 20행 이내)에서 "급여총액", "공제총액", "실지급액"과 같은 레이블을 검색하고, 같은 행의 우측에서 해당 금액을 찾습니다. 특정 고정 레이아웃(예: 제공된 Excel 이미지의 요약 정보 위치)에 대한 추가적인 검사 로직도 포함합니다.
-   **계산 검증**: `calculateSummary` 메소드는 추출된 `salaryItems`와 `deductionItems`의 `numericAmount`를 합산하여 총 지급액과 총 공제액을 별도로 계산합니다.
-   **최종 결정**: 직접 추출된 총액과 계산된 총액을 비교합니다. 직접 추출된 값이 0이 아니고 계산된 값과 지정된 허용오차(tolerance) 내에서 유사하면 직접 추출된 값을 사용하며, 그렇지 않으면 계산된 값을 우선합니다. 실지급액은 최종적으로 선택된 총 지급액과 총 공제액을 바탕으로 계산됩니다.

### 3.4. 패턴 매칭 활용

-   `salaryPatterns`, `deductionPatterns`: 광범위한 급여 및 공제 항목 이름을 인식하기 위한 정규표현식 배열입니다.
-   `amountPattern`: 금액으로 인정될 수 있는 문자열 형식(숫자와 콤마)을 정의합니다.
-   `summaryPatterns`: 요약 정보 레이블을 인식하기 위한 정규표현식입니다.
-   `isItemName`, `isAmount` 등의 헬퍼 메소드에서 이 패턴들을 사용하여 셀 내용의 유형을 판단합니다.

## 4. 확장성

### 4.1. 항목 패턴 업데이트

-   새로운 유형의 급여 항목이나 공제 항목을 지원해야 할 경우, `PayrollParser` 클래스 생성자 내의 `this.salaryPatterns` 또는 `this.deductionPatterns` 배열에 해당 항목을 인식할 수 있는 정규표현식을 추가하기만 하면 됩니다.

### 4.2. 명세서 레이아웃 변경 대응

-   **고정 블록 상수 조정**: 만약 주로 사용하는 명세서의 기본 구조(예: 급여 항목이 차지하는 행의 수, 이름 블록과 금액 블록 간의 간격 등)가 변경된다면, 클래스 상단에 정의된 `SALARY_NAME_ROW_COUNT`, `SALARY_AMOUNT_ROW_OFFSET` 등의 상수 값을 조정하여 대응할 수 있습니다.
-   **앵커 텍스트 변경**: 만약 "기본기준급"과 같은 핵심 앵커 텍스트가 변경된다면, `SALARY_ANCHOR_TEXT` 등의 상수를 업데이트합니다.
-   **(향후) 동적 전략 활성화**: 현재는 고정 블록 추출의 정확도에 집중하기 위해 동적/폴백 추출 로직(`extractItemsDynamically`, `fallbackItemExtraction`) 호출부가 주석 처리되어 있습니다. 만약 매우 다양한, 예측 불가능한 레이아웃의 파일을 처리해야 할 경우, 이 주석을 해제하여 파서의 유연성을 높일 수 있습니다. 이 동적 로직들은 힌트 텍스트를 기반으로 항목/금액 블록의 시작점을 찾으려고 시도합니다.

## 5. CSV 파일 처리

`parseCSV` 메소드는 CSV 파일을 텍스트로 읽어 각 줄을 분리하고, 쉼표(또는 다른 추측된 구분자)로 셀을 나누어 `ExcelSheetData`와 유사한 2차원 배열 형태로 만듭니다. 이 데이터는 `analyzeExcelStructure` 메소드로 전달되어 Excel 파일과 동일한 분석 로직(앵커 탐색, 고정 블록 추출 등)을 거치게 됩니다. 따라서 CSV 파일이 Excel 명세서와 유사한 표 구조를 가지고 있다면, 효과적으로 파싱될 수 있습니다.

## 6. 사용 방법 (코드 내)

```typescript
// 1. Parser 인스턴스 생성
const parser = new PayrollParser();

// 2. 파일 파싱 (비동기)
// file: File 객체 (HTML input[type="file"] 에서 얻은)
try {
  const parsedInternalData: ParsedInternalData = await parser.parseFile(file);

  // 3. 결과 포맷팅
  const result: FormattedPayrollResult = parser.formatResult(parsedInternalData);

  if (result.success && result.data) {
    // 성공: result.data (PayrollData 타입)를 사용하여 UI 업데이트
    console.log(result.data.employeeInfo);
    console.log(result.data.paymentDetails.salaryItems);
    // ... 등등
  } else {
    // 오류 처리
    console.error(result.error);
  }
} catch (e) {
  // 예외 처리
  console.error("파일 처리 중 심각한 오류:", e);
}
```

이 문서를 통해 `PayrollParser`가 특히 고정된 형식의 급여 명세서를 어떻게 정확하고 안정적으로 처리하는지에 대한 이해가 깊어지셨기를 바랍니다.
```
