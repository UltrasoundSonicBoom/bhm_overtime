# B9 부서 위키 — 아이디어 & 설계 초안

> 상태: 아이디어/계획 단계 → sub-spec 작성 진행 중  
> 작성: 2026-05-01  
> "부서" 개념: Schedule PRD의 `Team` 컬렉션 재사용 (`teams/{teamId}/wikiItems/{itemId}`). 별도 선행 조건 없음.

---

## 왜 중요한가

CareRole OS 135명 설문 결과:
- 운영지원·행정/총무 전체 pain point 1위
- 신규 직원 온보딩 3~6개월 → 담당자 의존
- 퇴직/이동 시 절차 소멸 → 매번 바닥부터 재구축
- 현재 해결책: 개인 Word 파일, 카톡 메시지 검색, 담당자 직접 문의

"문서화 없이 구전 전달"이 기본 상태. 병원 업무가 얼마나 복잡한지를 생각하면, 이건 조직 단위 리스크다.

---

## 핵심 아이디어

### 기능 한 줄 정의
"부서원이 아는 것"을 검색 가능한 형태로 만드는 도구.

### 해결하는 것
1. **신규 입사자 진입 장벽** — "이건 어디에 물어봐요?" → 위키 검색
2. **인수인계 공백** — 담당자 퇴직 후 지식 소멸 방지
3. **반복 질문 부하** — 수간호사/팀장이 매번 같은 걸 설명해야 하는 시간 낭비

### 해결하지 않는 것 (Non-goals)
- 병원 전사 지식 관리 시스템 대체 (너무 큰 범위)
- EMR·OCS와 연동 (접근 권한 없음)
- 실시간 업데이트 강제 (자율 기여가 전제)

---

## 두 가지 진입 경로

### 경로 A: "직접 쓰는 위키"
부서원이 자유롭게 절차를 작성·공유.
- Notion 같은 에디터
- 마크다운 + 이미지 첨부
- 항목별 담당자 태그

**장점:** 단순, 즉시 시작 가능  
**단점:** 아무도 안 쓸 수 있음 (자발적 기여의 함정)

### 경로 B: "기존 문서에서 추출" ← 추천
업로드한 문서(Word, PDF, HWP 등)를 AI가 구조화된 위키 항목으로 자동 변환.
- B6 마스킹 거친 문서만 입력 가능 (PII 제거 전제)
- "인수인계 문서 업로드 → 위키 항목 자동 생성"
- 기존에 있는 개인 Word 파일을 부서 공유 지식으로 변환

**장점:** "위키를 써야 한다"는 부담 없이 기존 문서를 변환  
**단점:** 문서 품질에 의존

### 경로 C: "대화로 추출" (장기)
"오늘 신규 직원에게 설명한 것을 위키로 저장하시겠어요?" 같은 AI 대화 후 자동 정리.

---

## 경로 B 상세 설계

### 사용자 흐름
```
1. 부서원이 Word/PDF/HWP 파일 업로드
2. B6 마스킹 파이프라인으로 PII 제거
3. AI가 문서에서 절차/지식 항목 추출
   - "제목", "담당자", "절차 단계", "주의사항", "관련 양식"
4. 자동 생성된 초안 미리보기
5. 부서원이 수정/승인
6. 부서 위키에 저장
```

### 위키 항목 구조
```
제목: [업무명]
업데이트: [날짜] by [담당자]
태그: [부서] [카테고리]

## 언제 사용하나
## 단계별 절차
  1. ...
  2. ...
## 주의사항
## 관련 양식/링크
## 문의처
```

### 검색
- 키워드 검색 (형태소 분석)
- 카테고리 필터 (인사/총무/원무/연구 등)
- "이런 경우엔?" 자연어 질문 → 관련 항목 반환

## "부서" 개념 정의

B9의 "부서"는 Schedule Tab PRD의 `Team` 컬렉션과 같은 개념을 사용한다.  
Schedule Phase 3 완료를 기다릴 필요 없다 — Team이 생성된 시점부터 B9 동작 가능.  
지금 당장 확정된 것:

| 결정 사항 | 옵션 A | 옵션 B |
|---------|--------|--------|
| 부서 단위 | Firebase Team (이미 PRD에 있음) | 별도 Department 엔티티 |
| 부서 생성 주체 | 팀 관리자만 | 누구나 (개인도 가능) |
| 위키 공개 범위 | 팀원 전체 | 부서 내 역할별 |
| B6와의 연계 | 마스킹 후만 허용 | 선택 사항 |

**권고:** Option A — Schedule PRD의 `Team` 컬렉션을 재사용.  
B9는 `teams/{teamId}/wikiItems/{itemId}` Firestore 경로에 얹는다.  
이렇게 하면 Schedule Phase 3 완료 전에도 팀 생성만 되면 B9 동작 가능.

## B9 vs B5 (반복 문서 초안) 관계

두 기능은 다른 곳을 건드린다:

| | B5 (반복 문서 초안) | B9 (부서 위키) |
|-|-------------------|--------------|
| 목적 | 개인의 반복 작업 줄이기 | 조직의 지식 보존 |
| 출력 | 개인이 쓸 문서 초안 | 부서 공유 위키 항목 |
| 수명 | 한 번 쓰고 끝 | 지속적으로 참조 |
| 사용자 | 개인 | 부서 전체 |

하지만 **연결점**이 있다:  
B5가 생성한 문서 초안 → "이걸 부서 위키로 저장할까요?" → B9 입력.  
B6 마스킹한 문서 → B9 위키 변환.  
**B6 → (B5) → B9가 자연스러운 흐름.**
