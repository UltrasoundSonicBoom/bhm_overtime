## 페르소나 분석

### Product Manager 관점

규정을 직접 읽고 동료가 못 받는 수당이 무엇인지 파악한 뒤, 그것을 계산하는 도구를 만든다.
pain-point가 조항 번호 단위로 정밀하다.

**강점**
- 내부자 관점의 문제 정의 — "인사팀에 또 물어봐야 하나"는 외부인이 쓸 수 없는 문장
- 신뢰를 기능으로 만든다 — "모든 데이터는 내 브라우저에만"을 온보딩의 전용 섹션으로 설계
- 규정 완전성 집착 — 빠진 조항은 사용자가 실제로 손해본다는 걸 안다
- 단계적 공개 — Phase 1-8로 기능을 순서 있게 쌓고 한 번에 다 만들려 하지 않음

**약점**
- 자기가 당사자가 아닌 기능에서 가정이 들어간다 (B1 폐기 교훈)

### Designer 관점

복잡한 것을 단순해 보이게 만드는 것이 아니라, **복잡한 것을 복잡한 채로 이해 가능하게** 만든다.

**강점**
- 컨텍스트가 함께 있는 UI — 수당 계산 옆에 조항 번호 인라인 표시
- 바텀시트·타임라인 등 패턴 일관성 — 한 번 배우면 전체에 적용됨
- 상태 명시 — 앱이 지금 어떤 상태인지 항상 사용자에게 알림
- 프라이버시 UX를 기능과 동급으로 설계

**약점**
- 정보 밀도가 높다 — 숙련 사용자에게 강하지만 첫 접속자에게 진입장벽

### AI Adoption 관점

AI를 도구 대체가 아니라 **정밀도 보완**으로 쓴다.

**강점**
- 계산은 결정론적으로, AI는 규칙이 깨지는 경계(비정형 파싱·PII 검증)에만 투입
- AI 출력을 검증한다 — `verifyPayslip`으로 파싱 결과를 계산값과 대조
- 로컬 우선 (Presidio → LLM opt-in), AI 기능에 사용자 동의 레이어 분리
- 개발 workflow 자체를 Claude Code subagent로 자동화

**약점**
- AI 실패 경로를 나중에 설계하는 경향 → 선행 설계 필요

## 개발 참조 지침

### 강점을 더 적극적으로 활용

**1. "내가 당사자다" 감각을 기능 설계의 출발점으로 삼는다**

- 기능 초안을 만든 직후, "나라면 이걸 이번 달 안에 실제로 쓰는가?"를 자문한다
- 동료에게 시연하기 전에 내가 먼저 쓴다

**2. 규정 조항을 UI 언어로 번역하는 것이 이 앱의 핵심 가치다**

- 수당·휴가·계산 관련 UI 문구에는 조항 번호가 기본값이다 (tooltip이 아니라 인라인)
- 규정이 바뀌면 UI 문구와 계산 로직을 동시에 수정한다. 계산만 바꾸고 설명을 두는 것은 버그다

**3. 계산의 신뢰성이 앱 전체의 신뢰성이다**

- 급여·퇴직금·수당 계산은 순수 함수로 유지한다. 네트워크·시간·사용자 상태에 의존하지 않는다
- 새 계산 함수를 추가하면 같은 날 단위 테스트를 작성한다. 테스트 없는 계산은 추측이다

---

### 약점을 구조적으로 보완

**4. 내가 당사자가 아닌 기능은 인터뷰가 선행한다**

- `docs/harness/persona-matrix.md` 해당 슬롯에 실제 인터뷰 내용이 있는지 확인한다
- 인터뷰 없으면 spec을 쓰지 않는다. 아이디어 메모로만 남긴다
- 인터뷰 대상은 "비슷한 상황의 동료 1명"으로 충분하다

**5. 정보 밀도가 높아질 때 "첫 화면 테스트"를 한다**

- "이 화면을 처음 본 사람이 30초 안에 하나의 행동을 할 수 있는가?"를 확인한다
- 숫자가 5개 이상 한 화면에 있으면 가장 중요한 숫자 1개를 히어로로 올리고 나머지는 보조로 내린다

**6. AI 기능은 실패 경로를 먼저 설계한다**

- spec에는 "정상 경로"와 "실패 경로"를 동등한 비중으로 작성한다
- AI 출력은 사용자가 수정할 수 있어야 한다. 수정 UI 없는 AI 기능은 미완성이다
- Presidio(로컬) → LLM(외부 opt-in) 순서를 지킨다

**7. 프라이버시 원칙이 마찰이 되지 않도록 한다**

- 로그인 없이도 핵심 기능 100% 동작은 지킨다
- 원칙 때문에 생기는 추가 단계는 최대한 자동화한다
- 동기화·공유 기능 추가 시 "이 데이터가 어디 저장되는가"를 UI 카피에 명시한다

## 기능 추가 판단 기준

기능을 추가할지 모를 때 이 순서로 판단한다:

```
1. 내가 이번 달에 실제로 쓸 기능인가?
   → YES: 만든다
   → NO: 2번으로

2. 동료 인터뷰에서 "이게 있으면 좋겠다"가 나왔는가?
   → YES: 만든다
   → NO: 아이디어 목록에만 남긴다 (만들지 않는다)
```
