## Pain Point별 실제 설계 상태

### 1위. 반복 문서·메일·협조전 과부하 (62%, 84/135명) — B5

**Pain Point 신규 정의:**  
"반복 데이터 입력"은 EMR이 핵심이 아니다. 병원 직원이 매일 처음부터 다시 쓰는 것은 세 가지다:

1. **문서작성** — 공문, 보고서, 기안서, 인수인계서, 환자 안내문, IRB 서류, 시설 점검 기록
2. **메일** — 같은 유형의 업무 이메일을 매번 처음부터 작성 (회신 포함)
3. **그룹웨어 협조전** — 병원 내 부서간 협조요청 문서 (병원 그룹웨어 전자결재 양식, 결재선 포함)

이 세 영역은 병원 전 직종에 걸쳐 나타나며, EMR과 무관하게 독립적으로 존재한다.

**페르소나 카드 (897건 채용공고, 2021-2026) 직종별 실태:**

| 직종 | 반복 작업 유형 | AI 활용기회로 명시 (원문) |
|------|-------------|----------------------|
| 간호직 (230건) | 인수인계서, 간호 기록 초안, 환자 안내문 | "간호 기록 초안 자동화", "인수인계 요약", "반복 환자 안내문 생성" |
| 연구지원/CRC | IRB 신청서, 과제보고서, 협약 문서 | "연구 문서 초안 자동화", "실험/과제 진행 현황 리포트 생성" |
| 간호조무사 (16건) | 환자 안내 스크립트, 인수인계 | "환자 안내 스크립트 자동화", "근무교대 인수인계 요약" |
| 약제직 | 복약 안내문, 조제 기록 | "반복 질의형 복약 안내 초안 생성" |
| 운영/시설 (4건) | 시설 점검 기록, 부서간 협조요청 | "시설 점검 이력 자동 정리" |
| 기타 행정 (84건) | 공문, 보고서, 그룹웨어 기안, 메일 | "반복 문서와 일정 정리 자동화" |

채용공고 전체 공통 요구사항: "문서작성/문서관리/문서기안 규정 및 지침에 대한 이해, 업무용 소프트웨어의 특성 및 기능 이해" — 모든 직종에 붙어있는 일반직 공통 요건.

**병원 그룹웨어 협조전 특이점:**
- 병원은 독자 그룹웨어(ERP 계열) 사용이 일반적이며, 부서간 협조요청은 표준화된 전자결재 양식
- 매번 같은 패턴의 요청서를 처음부터 작성 → LLM이 목적/수신부서/내용 입력받아 초안 생성 후 그룹웨어에 붙여넣기
- 메일과 협조전은 "같은 공지를 어떤 채널로 보내느냐"의 차이이므로 동일한 초안 생성 엔진으로 커버 가능

**실제 설계 상태: 없음 (0%)**
- B5: 이전까지 미할당 상태 → **이 문서에서 신규 할당**
- harness-design.md: B7="환자 설명문 생성기", B8="부서 AI 뉴스레터"로 이미 다른 슬롯 사용 중
- 문서·메일·협조전 자동화에 맵핑되는 슬롯 없었음

**필요한 설계 (B5 — 반복 문서·메일·협조전 AI 초안):**
- 페르소나: 간호직(인수인계서/환자 안내문) + CRC(IRB 요약) + 행정직(공문/협조전/메일) 3직종
- 기능 흐름:
  1. 작업 유형 선택 (문서 / 메일 / 협조전)
  2. 목적·수신 부서·핵심 내용 입력 (필드 최소화)
  3. LLM 초안 생성
  4. 인라인 편집 → 복사 or 다운로드
- 기술: Claude API + 직종별 프롬프트 템플릿 라이브러리 (EMR 연동 불필요)
- 난이도: ★★★ — 프롬프트 설계 + 간단한 UI. 즉시 시작 가능.

**결론: B5 슬롯을 "반복 문서·메일·협조전 AI 초안"으로 신규 할당. EMR 없이 바로 구현 가능.**

### 2위. 지식 소멸 / 인수인계 단절 — B9 (진행 중)

**Pain Point:**
- 담당자 퇴직·이동 시 절차 소멸 (구전)
- 신규 직원 온보딩 3~6개월 소요

**실제 설계 상태: 아이디어/계획 단계 (10%)**

| 항목 | 내용 |
|------|------|
| b-catalog.md | "🔲 미시작" — 페르소나 행 있음 |
| harness-design.md §4 | "B9: 부서 위키 자동 정리 (마스킹된 자료 → 부서별 위키)" 한 줄 |
| 아이디어 문서 | `docs/harness/b9-idea.md` — 경로 A/B/C, 부서 개념 정의, Firestore 경로 제안 |
| spec 파일 | 없음 (작성 필요) |
| plan 파일 | 없음 (작성 필요) |

**"부서" 개념 — Schedule Phase 3과 독립적으로 해결됨:**
- B9의 "부서" = Schedule PRD의 `Team` 컬렉션 재사용
- Firestore 경로: `teams/{teamId}/wikiItems/{itemId}`
- 팀 생성만 완료되면 B9 동작 가능 (Schedule Phase 3 완료 대기 불필요)

**필요한 설계:**
1. B9 sub-spec 작성 (Firestore 스키마, 위키 항목 구조)
2. AI 추출 프롬프트 설계 (업로드 문서 → 위키 항목 JSON)
3. B6(마스킹)과의 연계: 마스킹 후 자료만 위키에 들어감
4. UI 위치 결정 (새 탭 vs 찾아보기 탭 하위)

**결론: 아이디어 문서 완성, sub-spec 작성 단계로 진행.**

### 3위. 교대 피로 + 스케줄 공정성 불만 — B3 (부분)

**Pain Point:**
- B3이 다루는 것: 근무표 → 개인 달력 자동 입력 (OCR/파싱)
- B3이 다루지 않는 것: 야간 쏠림, 피로 누적, 불공정 배치 체감

**실제 설계 상태:**

| 컴포넌트 | 설계 수준 |
|---------|-----------|
| B3 (근무표→달력) | 페르소나 있음, spec/plan 없음 (5%) |
| 간호 근무 PRD | nurse-schedule-rule-workflow-prd.md 존재 (완성) |
| 공정성 대시보드 | 없음 (0%) |

**간호 근무 PRD 분석 (`2026-04-29-nurse-schedule-rule-workflow-prd.md`):**
- 범위: Team Plan (수간호사용 관리자 도구) — 개인 간호사 pain point 해결이 아님
- 다루는 것: import, 검증, 드래그앤드랍 수정, 배포, 교대 요청
- 다루지 않는 것: 야간 쏠림 히스토리, 개인별 피로 지수, 공정성 비교 리포트
- PRD의 Success Metrics에 "공정성" 지표 없음

**필요한 설계 (공정성 서브기능):**
- 대상: 개인 간호사 (B3 사용자), 수간호사 (PRD 관리자)
- 기능: 월별 야간 횟수·주말 횟수·연속 근무일 히스토리 조회, 팀 평균 대비 내 수치
- 데이터: Published schedule version 기반 집계 (PRD의 Firestore 모델 위에 올라탈 수 있음)
- 새 슬롯 필요 여부: B3 서브기능으로 가능, 또는 미할당 슬롯에 "B3-공정성" 신규 슬롯

**결론: PRD는 관리자 도구이고 개인 공정성 리포트 없음. 공정성 서브기능은 추가 설계 필요.**

### 4위. 급여/수당 불투명성 — B4 (미시작)

**Pain Point:**
- 명세서 항목 30개+, 약어 투성이
- 교대수당·심야가산 오계산 여부 확인 불가

**실제 설계 상태: 이름만 있음 (5%)**

| 항목 | 내용 |
|------|------|
| b-catalog.md | "🔲 미시작" — 페르소나 행 있음 |
| harness-design.md §4 | "B4: 명세서 → AI 분석 카드 (LLM 보조 파서 위)" 한 줄 |
| spec 파일 | 없음 |
| plan 파일 | 없음 |
| 기존 백엔드 | `backend/app/payslip_parser.py` — LLM 보조 파서 존재 (파싱만, UI 없음) |

**기존 자산 분석:**
- `backend/app/routes/payslip.py`: `/payslip/parse` 엔드포인트 — 명세서 이미지 → 항목 JSON 파싱
- 현재 UI: 급여탭 "명세서 업로드" → 파싱 결과 수동 확인 (사용자가 직접 해석)
- 미싱: 항목별 한글 설명, 법령 조항 연결, 전월 대비 차이 하이라이트

**필요한 설계:**
1. B4 spec 작성 (명세서 AI 분석 카드 UI)
2. 백엔드: 파싱 결과 → 각 항목에 "설명 + 근거 법령 + 교대 수당 공식" 매핑
3. 프런트: 분석 카드 컴포넌트 (항목명 + 설명 + 금액 + 전월 대비)
4. 데이터: 단협·SNUH 인사규정의 수당 명칭 매핑 테이블

**결론: 백엔드 기반은 있지만 B4 UI 설계는 0%. 파서 위에 "분석 카드" 레이어 신규 설계 필요.**
